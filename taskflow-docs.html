Introduction

Project Overview

TaskFlow is a browser-based task management application built entirely with HTML, CSS, and vanilla JavaScript — no backend, no database, no build tools required.

It allows users to create, read, update, and delete personal tasks. Each task carries a title, description, due date, time, priority level, and status. All data is stored in the browser's localStorage, meaning tasks persist across page refreshes without any server infrastructure.

The app is structured as a Single Page Application (SPA) — not through a framework like React or Vue, but through a custom routing pattern that fetches HTML fragments and injects them into a persistent shell layout.

🎯
Purpose
Help individuals manage daily to-dos and projects. Organize tasks by status, priority, and due date with a distraction-free interface.
👤
Target Users
Students, professionals, and anyone needing a lightweight personal task manager that works offline and needs no account or login.
⚙️
How It Runs
Open index.html in any browser. No server, no install, no dependencies to download. Works completely offline once loaded.
💾
Data Storage
All tasks are saved to the browser's localStorage as a JSON array. Data persists through refreshes but is local to each browser/device.

Features

All Features

TaskFlow ships with 12 distinct features. Here's every one of them and exactly how each works internally.

➕
Create Task
Form with title*, description, due date*, time, status*, and priority. Validates required fields before calling addTask(), which pushes a timestamped object to localStorage.
✏️
Edit Task
Edit button stores the task ID in localStorage as 'editingTaskId'. The form page detects this, pre-fills all fields, and calls updateTask() instead of addTask() on submit.
🗑️
Delete Task
Triggers a confirmation dialog before deletion. On confirm, deleteTask() filters the task from the array. The card is removed without a full re-render.
✔️
Toggle Completion
Clicking the circular checkbox on any card flips its status between 'done' and 'pending'. The card re-renders with a strikethrough title and green checkmark if completed.
🔄
Quick Status Buttons
Each task card has three emoji buttons (⏳ 🔄 ✅) for rapid status changes. Clicking any calls updateTask() and re-renders the task list instantly.
🔍
Real-Time Search
The search input fires on every keystroke. It loops through rendered task cards and toggles display:none based on whether the title text includes the search query.
🏷️
Status Filtering
Five filter tabs (All, Completed, In Progress, Pending, Overdue) filter the task list client-side. No re-fetch — cards are shown/hidden instantly by matching data-status attributes.
⚠️
Overdue Detection
computeStatus() compares each task's due date against today. Tasks with a past due date that aren't marked 'done' are automatically classified as 'overdue' and styled in red.
📊
Live Home Stats
The homepage shows real-time counts for Total, Completed, In Progress, and Overdue tasks. updateStats() recalculates from localStorage every time the Home page is visited.
🌙
Dark Mode
Toggles the data-bs-theme attribute on <html>. CSS variables respond automatically. Preference is saved to localStorage and restored on every page load.
📱
Responsive Design
Bootstrap's grid system and a Bootstrap offcanvas component handle mobile navigation. The layout adapts from a full desktop nav to a hamburger-triggered slide-in menu.
🔔
Toast Notifications
showToast(message, type) uses Bootstrap's Toast API for non-blocking feedback. Four types: success (green), danger (red), warning (amber), info (blue). Auto-dismisses after 3 seconds.

Code Explained

Storage Layer

All CRUD operations go through four thin wrapper functions that abstract localStorage. Every function reads or writes the same 'tasks' key.

JavaScript script.js — Storage Functions
// Read: always parses fresh from localStorage function getTasks() { return JSON.parse(localStorage.getItem('tasks')) || []; } // Write: serializes entire array back to localStorage function saveTasks(tasks) { localStorage.setItem('tasks', JSON.stringify(tasks)); } // Create: generates a unique ID via timestamp function addTask(task) { const tasks = getTasks(); task.id = Date.now(); // timestamp = unique number tasks.push(task); saveTasks(tasks); } // Delete: filters target task out of array function deleteTask(id) { let tasks = getTasks(); tasks = tasks.filter(t => t.id !== id); saveTasks(tasks); } // Update: merges new data into matching task function updateTask(id, updatedTask) { const tasks = getTasks(); const index = tasks.findIndex(t => t.id === id); if (index !== -1) { tasks[index] = { ...tasks[index], ...updatedTask }; saveTasks(tasks); } }
📌
Why getTasks() reads fresh every time: Rather than caching the array in a variable, getTasks() always parses from localStorage. This ensures that if two tabs are open, both see consistent data. The trade-off is a small JSON.parse on every call, which is negligible for typical task counts.

Each task object stored looks like this:

JSON localStorage['tasks'] — Task Object Shape
{ "id": 1715000000000, // Date.now() timestamp "title": "Prepare weekly report", "desc": "Include Q2 metrics and next steps", "date": "2026-05-15", // YYYY-MM-DD from <input type="date"> "time": "14:30", // HH:MM, optional "status": "pending", // "pending" | "progress" | "done" "priority": "high" // "low" | "medium" | "high" }
Code Explained

Page Router

The SPA-style routing is the most technically interesting part of the codebase. loadPage() handles fetching, script re-execution, transitions, and nav highlighting.

JavaScript script.js — loadPage()
function loadPage(page, activeLinkId) { const content = document.getElementById('main-content'); // Step 1: fade out the current content content.style.opacity = '0'; content.style.transform = 'translateY(6px)'; content.style.transition = 'opacity 0.18s ease, transform 0.18s ease'; // Step 2: fetch the HTML fragment file fetch(page) .then(res => res.text()) .then(html => { setTimeout(() => { // Step 3: inject HTML into <main> content.innerHTML = html; // Step 4: re-execute any <script> tags in the fragment // (innerHTML does NOT auto-run scripts, so we clone them) content.querySelectorAll('script').forEach(old => { const s = document.createElement('script'); s.textContent = old.textContent; document.body.appendChild(s); old.remove(); }); // Step 5: fade in, scroll to top, re-bind links content.style.opacity = '1'; content.style.transform = 'translateY(0)'; attachInternalLinks(content); window.scrollTo({ top: 0, behavior: 'smooth' }); }, 180); // 180ms matches the fade-out duration }); // Step 6: highlight the active nav link if (activeLinkId) setActiveLink(activeLinkId); }
🔑
Why clone scripts? When you set innerHTML, any <script> tags in the new HTML are parsed but never executed by the browser. To run them, the code creates a fresh document.createElement('script'), copies the source code into it, then appends it to the <body> — this triggers execution. The old inert script is then removed.

Task Rendering

renderTasks() is the view layer. It reads all tasks, builds card HTML for each, appends them to the DOM, then wires up all interactive buttons.

JavaScript script.js — renderTasks() core loop
function renderTasks() { const container = document.getElementById('task-list-container'); const tasks = getTasks(); container.innerHTML = ''; // clear before rebuild tasks.forEach(task => { // Compute effective status (handles overdue detection) const status = computeStatus(task); const card = document.createElement('div'); card.className = `tf-task-card ${status}`; // CSS class drives color bar card.dataset.status = status; // data attr used for filtering card.innerHTML = ` <div class="tf-task-check" data-id="${task.id}">...</div> <div class="tf-task-body"> <div class="tf-task-title">${task.title}</div> <div class="tf-task-meta">...date, time, priority...</div> </div> <div class="tf-task-actions"> <button class="status-btn" data-id="${task.id}" data-status="pending">⏳</button> <button class="edit-btn" data-id="${task.id}">✏️</button> <button class="delete-btn" data-id="${task.id}">🗑️</button> </div>`; container.appendChild(card); }); attachTaskActions(); // bind click events AFTER DOM is updated }

Form Logic

The create-task form serves two purposes: creating new tasks and editing existing ones. It detects which mode it's in by checking localStorage for an editing ID.

🆕 Create Mode
When the form loads with no editingTaskId in localStorage, it shows "New Task" and an "Add Task" button. On submit, addTask() is called and editingTaskId is never touched.
✏️ Edit Mode
When editingTaskId exists in localStorage, the form pre-fills every field from the matching task object. The button changes to "Update Task". On submit, updateTask() is called and editingTaskId is cleared.
JavaScript create-task.html — Edit mode detection
// Runs when the create-task page loads const editingId = localStorage.getItem('editingTaskId'); if (editingId && window.getTasks) { const task = getTasks().find(t => t.id == editingId); if (task) { // Pre-fill every form field with the task's existing values document.getElementById('taskTitle').value = task.title; document.getElementById('taskDate').value = task.date; document.getElementById('taskStatus').value = task.status; // Change the button label to signal edit mode document.querySelector('.tf-submit-btn').innerHTML = '<i class="bi bi-save"></i> Update Task'; } }

Dark Mode System

Dark mode is entirely CSS-driven. JavaScript only toggles an attribute — the stylesheet handles all the visual changes automatically through CSS variable overrides.

JS Side — Toggle Logic
On button click, the current data-bs-theme attribute is read. If 'dark', switch to 'light' and vice versa. Save the result to localStorage as 'tf-theme'. On page load, restore from localStorage (default: 'light').
CSS Side — Variable Overrides
The styles.css file has a [data-bs-theme="dark"] block that overrides --tf-bg, --tf-surface, --tf-border, --tf-ink, and shadow values. Every component using these variables automatically updates — no JS class toggling needed.
Quality

Bugs Fixed

Three bugs were identified in the original codebase and corrected in the improved single-file version.

🔴 Bug 1 — Duplicate Toast After Edit
In create-task.html, the form submit handler calls showToast('Task added successfully!', 'success') unconditionally at the bottom of the handler — after the conditional block that already shows the correct message. This means even after editing a task, the user sees "Task added successfully!" instead of "Task updated successfully!"
✅ Fix — Removed Duplicate Toast Call
The redundant showToast() call after the if/else block was removed. The conditional now exclusively controls which message is shown — "Task added successfully!" for create, and "Task updated successfully!" for edits.
🔴 Bug 2 — Hardcoded Home Stats
The homepage stat cards display the numbers 12, 7, 3, 2 hardcoded in the HTML. These never change no matter how many tasks the user creates, edits, or deletes. The stats are decorative rather than functional.
✅ Fix — Live Dynamic Stats
A new updateStats() function reads the current task array and counts tasks per computed status. It's called every time the Home page is shown, so the numbers always reflect real data.
🔴 Bug 3 — Overdue Filter Non-Functional
The task list has an "Overdue" filter tab and the CSS styles for .tf-task-card.overdue exist. However, no code ever sets a task's status to "overdue". Tasks past their due date remain "pending" or "progress" forever — the overdue tab always shows zero results.
✅ Fix — computeStatus() Overdue Detection
A computeStatus(task) function was added. It compares the task's due date against today's date. If the task is not 'done' and the due date is in the past, it returns 'overdue'. This is used at render time — the actual status in storage is not mutated.
🔴 Bug 4 — XSS via innerHTML
Task titles and descriptions entered by the user are inserted directly into card HTML using template literal strings and innerHTML. A title like <img src=x onerror="alert(1)"> would execute JavaScript.
✅ Fix — HTML Escaping via escHtml()
An escHtml(str) function escapes &, <, >, and " characters before user content is injected into the DOM. All title and description values pass through this function before rendering.
Quality

Suggested Improvements

Beyond the bugs fixed, here are recommended enhancements for making TaskFlow more robust, scalable, and user-friendly.

⚡ Performance — Virtual Rendering
Currently renderTasks() rebuilds all card DOM nodes on every status change. With hundreds of tasks this becomes slow. A smarter approach would only update changed cards (DOM diffing) or use a virtual scroll list for large datasets.
🔒 Security — Content Security Policy
The inline script execution pattern used by the router (cloning and appending <script> elements) is blocked by strict CSP headers. Production deployments should use event-driven architecture instead of inline script injection for better security posture.
📂 Code Organization — Module Split
All 300+ lines of logic live in one script.js file. Splitting into modules — storage.js, render.js, router.js, ui.js — would make the codebase significantly easier to maintain, test, and extend as the feature set grows.
♿ Accessibility — ARIA Roles
Task cards and status buttons lack ARIA attributes. Adding role="listitem", aria-label, and aria-live regions for toast notifications would make the app usable with screen readers and keyboard-only navigation.
🔄 Scalability — IndexedDB
localStorage is limited to ~5MB per domain and only stores strings. For a production task manager, IndexedDB offers structured queries, much larger storage limits, and better performance for bulk operations.
🏷️ Feature — Task Categories / Tags
The data model could be extended with a tags or category array, enabling multi-dimensional filtering (e.g., Work + High Priority). The card UI and filter tabs would need to reflect this added dimension.
🎯
Overall Assessment: TaskFlow is a well-structured beginner-to-intermediate project. The CSS custom property theming system is particularly polished — dark mode support is seamless, the color palette is cohesive, and the card design is clean and professional. The core SPA routing pattern is clever and works well for a small app. With the four bugs fixed and a single-file refactor, it is fully production-ready for personal use.
